# Game Project Lifecycle

Use this index to pick the right workflow. Read the linked skill for step-by-step guidance.

| Phase | Skill | When to use |
|-------|-------|-------------|
| Scope | `scope-game-milestone` | Defining prototype, alpha, or milestone scope and cut list |
| Bootstrap | `bootstrap-game-project` | Starting a new game repo or scaffolding core systems |
| Feature | `game-feature-workflow` | Adding mechanics, levels, UI flows, or gameplay systems |
| Run / build | `run-game-project` | Editor play mode, dev builds, tests, export commands |
| Playtest | `run-playtest-session` | Structured milestone playtest (feel, balance, holistics) |
| Debug | `debug-gameplay` | Reproducing and fixing a specific gameplay bug |
| Performance | `optimize-game-performance` | Frame time, load time, or memory regressions |
| Polish | `polish-game-feel` | Juice, feedback, and tuning when mechanics already work |
| Release | `ship-game-release` | Feature freeze, build, verify, publish a milestone |
| Hotfix | `patch-game-hotfix` | Minimal post-launch fix or balance patch |

## Rules (always consult)

- `game-production` — milestone scope and cut-list discipline
- `game-dev-conventions` — coding and asset habits
- `game-architecture` — state ownership and system boundaries
- `game-release` — versioning and ship checklist (near release)

## Stack child

For engine-specific steps (scenes, export presets, GDScript), use the repo's child layer skill (e.g. `godot-scene-workflow`).
